# ADR-003: ✅ フロー情報の置き場としてnotes/ディレクトリを設け、Issueコメントと併用する

- **日付**: 2026-03-14
- **ステータス**: ✅ 承認済み
- **チケット**: #27

## コンテキスト

ドキュメントにはストック情報（時間が経つほど価値が高まる）とフロー情報（作業中に価値を持ち、時間経過で価値が消える）の2種類がある。

`docs/spec/` や `docs/decisions/` はストック情報として整備されていたが、フロー情報（作業ログ・スパイク調査メモ・たたき台・議論ログ）の置き場が未定義だった。その結果：
- 作業中の判断経緯や調査内容が揮発し、セッション再開時にコンテキストを再現しにくい
- Issueコメントに詳細ログを書くと読みにくくなる
- ストック情報にフロー的な内容が混入するリスクがある

## 検討した選択肢

### 選択肢A: notes/ディレクトリを設けてIssueコメントと併用する
- **pros**: フロー情報の置き場が明確になる。Issueコメントは要旨のみにできて読みやすさを保てる。リポジトリ内で完結する
- **cons**: ディレクトリが一つ増える

### 選択肢B: Issueコメントに集約する
- **pros**: 既存運用の延長で新しいディレクトリが不要
- **cons**: Issue単位に縛られるためチケット横断の情報が置きにくい。詳細ログを書くとコメントが読みにくくなる

### 選択肢C: GitHub Discussionsや外部ツールを使う
- **pros**: 議論・ログ用途に特化したUIが使える
- **cons**: リポジトリの外にコンテキストが分散する。個人開発では過剰

## 決定

**選択肢Aを採用する。**

### 理由

Issueコメントの読みやすさを保ちつつ、チケット横断の情報も一元管理できる。外部ツールへの依存なくリポジトリ内で完結するのが個人開発には適切。ディレクトリが増えるデメリットはフロー/ストックの分離という恩恵に対して小さい。
