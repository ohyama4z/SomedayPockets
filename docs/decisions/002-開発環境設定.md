# ADR-002: ✅ 開発ツールとしてnpm + Vitest + Playwrightを採用する

- **日付**: 2026-03-14
- **ステータス**: 承認済み
- **チケット**: #15

## コンテキスト

プロジェクト初期セットアップにあたり、パッケージマネージャー・テストフレームワークを決定する。
ローカル開発はWrangler（ローカルモード）を使う前提が既に決まっている。

## 検討した選択肢

### パッケージマネージャー

#### 選択肢A: npm
- **pros**: Node.jsに標準付属で追加インストール不要。最も馴染みがある
- **cons**: pnpmと比べてインストール速度・ディスク効率が劣る

#### 選択肢B: pnpm
- **pros**: ハードリンクによるディスク節約・高速インストール。ゴースト依存を防ぐ厳格な依存解決
- **cons**: npmと比べて馴染みが薄い。個人開発では差が出にくい

#### 選択肢C: bun
- **pros**: 爆速インストール・ランタイム兼用
- **cons**: Cloudflare系ツールとの相性がまだ不安定なケースあり。成熟度が低い

### ユニットテスト

#### 選択肢A: Vitest
- **pros**: ESMネイティブで高速。Next.js/TypeScript構成との相性が良い。設定が軽い。Viteエコシステムと統一感がある
- **cons**: Jestと比べて歴史が浅い（ただし実用上の問題はほぼない）

#### 選択肢B: Jest
- **pros**: 長年の実績。ドキュメント・事例が豊富
- **cons**: ESM対応が後付けで設定が重い。VitestよりもTypeScript/ESMプロジェクトで扱いにくい

### E2Eテスト

#### 選択肢A: Playwright
- **pros**: Next.js公式サポート。並列実行・複数ブラウザ対応。現在のE2Eデファクトスタンダード
- **cons**: Cypressと比べてUIが地味（ただしCIでは関係ない）

#### 選択肢B: Cypress
- **pros**: UIが使いやすくデバッグしやすい
- **cons**: 重い。並列実行は有料プラン。Playwrightと比べて機能・速度で劣る

## 決定

**npm + Vitest + Playwrightを採用する。**

### 理由

- **npm**: 個人開発ではpnpmのディスク効率・速度の利点が小さく、馴染みのあるnpmで十分
- **Vitest**: ESMネイティブで設定が軽く、このプロジェクトの構成（Next.js + TypeScript）との相性が最も良い。Jestの設定の重さを避けたい
- **Playwright**: Next.js公式サポートがあり、CIでの利用実績も豊富。Cypressの有料制約を避けたい
