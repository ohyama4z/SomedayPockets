# オレオレTodoリスト — 要件定義書

## このドキュメントの目的

このドキュメントは「オレオレTodoリスト」のシステム要件（What）を記述する。
背景となるユーザーの思考特性（Why）は [思考特性マップ](2026-03-26_思考特性マップ.md) を参照。
実装者はWhyを深く理解した上でWhatの判断に迷ったときにWhyに立ち返ること。

---

## 設計思想

このシステムは「掌握の鏡」である。ユーザーの頭の中にある掌握感を外部に写し取り、認知負荷を下げながら全体を把握できる状態を維持する。ユーザーが能動的にやるのは「捕獲」と「朝の組み立て（弾選び）」だけ。それ以外はシステムが回す。

## 三層構造

### Layer 1: 捕獲ボックス（入口）
- **役割**: 思いついた完了状態をパッと投げ込む場所
- **UX要件**:
  - 最小限の入力で捕獲できること（タイトルだけでOK）
  - 丁寧さを強制しない。メタデータの入力は任意
  - いつでもどこでもすぐ書ける（モバイルファースト）
- **捕獲時の任意メタデータ**:
  - 種別: want（やりたい）/ must（やるべき）
  - 即弾フラグ: 今すぐできるものは熟成庫をスキップして弾に直行

### Layer 3: 熟成庫（中枢）
- **役割**: 捕獲されたタスクが沈んでいる場所。普段は見ない。システムの心臓部。
- **基本動作**:
  - 捕獲されたタスクはデフォルトでここに入る（即弾フラグ付きを除く）
  - ユーザーが自分から棚卸しする必要はない
  - システムが適切なタイミングでタスクを「浮上」させる
- **浮上条件**（タスクが熟成庫からLayer 2に押し出される条件）:
  - **期限接近**: 期限が設定されているタスクが近づいたとき
  - **鮮度アラート**: 一定期間放置されたタスクが腐る前に浮上。「やりたい」で入ったものが「やだ」に劣化する前に気づかせる
  - **外部トリガー**: 待ちタスクの条件が満たされたとき（ユーザーが手動で「トリガー来た」と叩く、または将来的に自動検知）
  - **「いつでもいい義務」の定期浮上**: 毎回判断コストを払わせるのではなく、システムが「今週これどう？」と低頻度で差し出す
- **掌握のための全体ビュー**:
  - 普段は見ないが、見たいときにはすべてのタスクを一覧できる
  - 漬物石の数や鮮度状態がざっくり把握できる（不安を消すための鏡）

### Layer 2: 今日の弾（出口）
- **役割**: 組み立て済みの「今日やること」リスト
- **UX要件**:
  - やりたいこと / やるべきことの区別が視覚的にわかる
  - 主旋律（やりたいこと）が目立ち、やるべきことはそこに溶け込む見え方
  - 隙間時間に見返したとき、すぐ手が動かせるものだけが並んでいる
  - ノイズゼロ。今日やれないものは一切見えない
- **組み立てモード**:
  - 熟成庫から浮上してきたもの + 自分で引き上げたものを、今日の弾に配置する
  - 順序を自分で決められる（連鎖の設計 = 快の源泉）
  - 前日夜に組み立てておくことも、当日朝にその場で組み立てることも両方できる
- **実行中の体験**:
  - 完了したら消える（達成感）
  - 並列で走っているもの（洗濯機回し中、AI調べ中など）のステータスが見えると万能感が増す（将来的にnice-to-have）

## タスクのライフサイクル

```
捕獲 → [即弾？] → Yes → 今日の弾
              → No  → 熟成庫 → [浮上条件発動] → 今日の弾 → 完了
                              → [ユーザーが引き上げ] → 今日の弾 → 完了
```

## タスクの属性

| 属性 | 必須/任意 | 説明 |
|------|-----------|------|
| タイトル | 必須 | 完了状態を表す短い文（例:「福岡宿をとる」） |
| 種別 | 任意 | want / must。未指定でもOK |
| 期限 | 任意 | ある場合のみ。浮上条件に使う |
| 鮮度 | 自動 | 捕獲日時から自動計算。経過日数に応じて劣化表示 |
| トリガー条件 | 任意 | 待ちタスクの場合、何が起きたら実行可能になるかのメモ |
| ステータス | 自動 | 熟成中 / 浮上済み / 今日の弾 / 完了 |

## 絶対に守るべき原則（実装者向け）

1. **捕獲の摩擦をゼロにせよ** — タイトル一行で投げ込めること。メタデータ入力を強制したら使われなくなる
2. **朝一のビューはノイズゼロ** — 今日の弾と浮上してきたサジェストだけ見える。漬物石は見えない
3. **倉庫の棚卸しをユーザーにさせるな** — 鮮度管理・浮上判断はシステムの仕事
4. **やりたい/やるべきの区別を消すな** — 混ぜた瞬間にリストがテンションを削る装置になる
5. **組み立ての自由を奪うな** — 順序や構成はユーザーが決める。自動スケジューリングはNG。あくまで素材を差し出す側
6. **掌握できていることを常に保証せよ** — 見たいときには全量見られること。「システムが隠しているものがあるかも」という不安を与えない

## 未決事項（要検討）

- 浮上条件の具体的な閾値（鮮度は何日で警告？）
- want/mustの判定をユーザーに聞くか、後からでも変えられるようにするか
- 外部トリガーの入力方法（手動のみ？通知連携？）
- プラットフォーム（モバイルアプリ？PWA？既存ツールの拡張？）
- 並列実行のステータス表示の要否（nice-to-haveだが優先度は低い）
